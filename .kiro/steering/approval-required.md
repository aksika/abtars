# Approval Required Before Changes

If you spot boilerplate, duplication, or a cleaner solution during implementation — suggest the refactoring to the user before doing it. Do not silently refactor.

- Present the proposed change
- Wait for approval
- Only then implement

This applies to ALL code changes — no exceptions.

## Planning vs Implementation

When the user asks a question, requests clarification, or discusses a plan — that is still planning, not a "go" signal. Do NOT start writing or modifying code until the user explicitly says to implement (e.g. "go", "implement", "do it", "ship it").

- If the user asks "what about X?" after a plan seems finalized — answer the question, do NOT start coding
- If the user says "let's plan" or "thoughts?" — stay in discussion mode
- If the user reviews a plan and asks follow-up questions — they are still deciding, wait
- Only an explicit implementation instruction transitions from planning to coding
